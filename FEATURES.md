## 5. 기능 상세

### 5.1 F-01 공고 올리기
- **목적:** 카톡방에 흩어진 단발 촬영 공고를 한 곳에 모은다. 양식은 현장 컨벤션 그대로.
- **플로우:** `/jobs/new` → 구조화 필드 입력 → 자동 조립 미리보기 확인 → 등록 → `/jobs/[id]`
- **상세:** 📄 `specs/F-01-job-posting.md`

### 5.2 F-02 카카오톡 소셜로그인
- **목적:** 카카오 OAuth 한 번으로 로그인. Phase 1 은 스텁(이름 입력) — 키 발급 후 같은 인터페이스로 교체.
- **플로우:** `/login` → "카카오로 시작" → 이름 입력 → 매칭 시 로그인 / 비매칭 시 "초대 링크 필요"
- **상세:** 📄 `specs/F-02-auth.md`

### 5.3 F-03 가입 인증(지인 추천만 가입가능)
- **목적:** 지인 추천 기반 신뢰형 가입. 기존 회원의 초대 토큰 없이는 가입 불가.
- **플로우:** 회원이 `/me` 에서 초대 링크 발급 → 공유 → 신규는 `/invite/[token]` 에서 가입 폼 → 가입 완료 → 자동 로그인
- **상세:** 📄 `specs/F-02-auth.md` (F-02 와 통합 spec)

### 5.4 F-04 평점·평판 시스템 (+ 신청 흐름)
- **목적:** 함께 일한 양측이 별점 1~5 + 한 줄 후기를 남겨, 모르는 사이에서도 신뢰 판단 가능.
- **플로우:** 신청 제출 → 촬영 종료(`shootDate` 지남) → 신청자/작성자 서로 평점 → 공고 카드 작성자 옆 별점 노출 + 프로필 상세
- **상세:** 📄 `specs/F-04-ratings.md` (블랙리스트는 폐기됨)

> 복잡한 기능은 요약만. 상세는 `specs/<feature>.md`.

## 6. 데이터 모델 (Phase 1 — 인메모리 스텁 기준)

```
users(id, kakao_id?, name, role, created_at)
  └─ invites(id, code, issued_by → users.id, used_by? → users.id, expires_at, used_at?)
  └─ jobs(id, author_id → users.id, role, shoot_date, headcount, gender_pref,
          pay, transport, call_time, location, end_time, notes,
          apply_fields[], status, created_at)
  └─ blacklist(id, owner_id → users.id, target_name, target_contact, reason, created_at)
```

상세 컬럼 정의는 각 spec 참조 (`specs/F-01~F-04`).

- 권한 정책 (향후 Supabase RLS):
  - `jobs`: open 상태는 누구나 read, write 는 본인만
  - `invites`: 발급자 본인만 read, 토큰 자체는 공개 URL
  - `blacklist`: 본인이 등록한 항목만 read/write (개인 블랙리스트)
